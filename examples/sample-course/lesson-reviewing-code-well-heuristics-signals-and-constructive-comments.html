Reviewing Code Well: Heuristics, Signals, and Constructive Comments

Why a repeatable review process matters

Good reviews are not about catching every defect. They are about raising the quality of changes in a predictable, respectful way, and helping the team move faster over time. A repeatable process keeps your attention on what matters, avoids nitpicking, and makes your feedback easier to act on.

A solid process answers three questions:

  • What do I look at first, and why?
  • How do I decide what to block and what to suggest?
  • How do I write comments that people can use?

This lesson walks through a practical sequence you can apply to any code change, the signals that should make you slow down or ask for help, and the tone and structure of comments that lead to improvement without friction.

Review the change the author made, not the code you wish they had written.

Start with context before reading code

Context helps you review the right thing. Without it, it is easy to argue over style while missing a broken assumption or a mismatched interface. Before diving into lines of code:

  • Read the change description. Look for the problem statement, scope, what is out of scope, and how the author validated the change. If the description is thin, ask for more context before reviewing in detail. Example: “What user flow is this affecting, and what are the acceptance criteria?”

  • Skim the diff map. Most tools show a list of files and an approximate size of changes. Use this to plan where to start and to assess risk. Example: “One migration, two model files, one controller, a dozen test files.” That suggests data shape changes, so you will look for migration safety and test coverage.

  • Check linked issues, designs, and specs. If there is a ticket number, click it. If the code references a design doc, open it. You do not need to read every word, but you should verify alignment. Example: “Spec says 5 percent throttle; the code sets 50 percent.”

  • Run the code or tests if needed. If the change affects behavior you can observe quickly (a CLI, an endpoint, a small UI), run it locally or in a preview environment. If the author linked to a staging URL or included a test plan, follow it. If it will take more than a few minutes, timebox and decide whether your run will uncover value or if you can defer to automated checks or a later pass.

  • Scan for risk upfront. Are there migrations, security-related files, shared libraries, or public APIs changed? This helps you choose your review order and depth.

Build a simple habit: write a quick intent check before detailed line notes. For example: “I understand this change aims to streamline OAuth token refresh by caching tokens per user session and adding retry on 401. I plan to review the interface of TokenCache first, then tests, then performance. Let me know if I missed anything.” This aligns expectations and invites correction early.

A practical review order

Jumping into the first file often leads to scattered comments. A consistent order makes your review faster and clearer. A reliable sequence:

  1. Design and interfaces. Are modules and functions arranged sensibly? Does the change align with the architecture and API boundaries? Are names and responsibilities clear?
  2. Correctness and tests. Does the code do what the description claims? Do tests cover the main path and failure modes? Are tests readable and reliable?
  3. Edge cases. What happens at boundary conditions? Empty inputs, timeouts, missing permissions, retries, idempotency.
  4. Security and performance. Any injection risks, unsafe parsing, exposed secrets, or slow paths introduced by the change? Memory and concurrency concerns.
  5. Readability and maintainability. Naming, structure, duplication, comments, and adherence to style guides.

This order moves from big decisions to fine details. If the design is off, there is no point debating variable names. If correctness is shaky, performance nits are premature.

Here is what this looks like in practice:

  • Design and interfaces example:

    • Original: a feature flag check is baked into a low-level database helper. That couples data access to feature rollout policy.
    • Comment: “Can we keep rollout logic at the service layer and pass a boolean intent into the repository? This keeps the repository reusable and testable.”

  • Correctness and tests example:

    • The change adds retry-on-timeout but only tests the success case.
    • Comment: “Do we have a test that simulates timeout on the first attempt and verifies backoff timing and max attempts?”

  • Edge cases example:

    • A new endpoint accepts an optional filter but does not validate unknown filters.
    • Comment: “What should happen if clients send an unsupported filter? Should we return 400 to avoid silent ignore?”

  • Security and performance example:

    • A log line includes request headers. That might leak Authorization.
    • Comment: “Headers can include secrets. Can we mask or whitelist instead of dumping all headers?”

  • Readability example:

    • A 60-line function mixes parsing, validation, and I/O.
    • Comment: “Could we split parseConfig, validateConfig, and applyConfig? It should make testing and future changes easier.”

Spotting risk signals early

Not every change deserves the same depth. Risk signals help you decide when to slow down, escalate, or ask for the change to be split.

  • Very large diffs. A 2,000-line PR across 40 files is easy to misreview. It often hides unrelated changes. Ask to split by concern. Example comment: “This includes API changes, UI refactor, and test infra updates. Could we separate API contract changes into their own PR and keep this one to UI behavior? That will make test review and rollout safer.”

  • Low or missing tests. A change to core logic with no new tests is risky. Concrete approach: check coverage of new branches. Example: “parseDate now handles time zones and DST; I do not see tests for DST transitions or invalid offsets. Could you add cases for ‘2024-03-10 02:30’ and ‘UTC+14’?”

  • Duplicated code. Copy-paste across modules creates maintenance risk. Example: “The validation for email appears in three handlers. Can we centralize to validateEmail in common/validators and import it here? That will reduce drift.”

  • Deep nesting and complexity. Five levels of if/else with multiple early returns is a bug magnet. Example: “This function has cyclomatic complexity 18. Could we extract the discount type handlers into a map of strategies to flatten control flow?”

  • Undocumented changes. Files touched without explanation, such as config toggles, CI scripts, or security settings, can have outsized impact. Example: “I see we changed S3 bucket policy to public-read. Can you explain the intent and add a note to the rollout plan with a verification step?”

Other subtle signals:

  • Editing generated files. Did the author commit build artifacts or lockfiles by hand?
  • Changing shared interfaces. Are there downstream consumers notified or tested?
  • Removing logs or metrics. Does this affect observability or alerts?

When you see two or more signals together (e.g., large diff and low tests), treat it as high risk. Pause, align on scope, and decide on a plan: split, add tests first, stage rollout behind a flag, or involve a second reviewer.

Run code and tests when it adds value

You do not need to run every change locally. Be selective:

  • Run if behavior is hard to reason about from code alone, or if the author requests specific validation steps.
  • Run if UI or CLI flows are central to the change, and a quick smoke test can catch obvious miswires.
  • Run if you suspect environment or config issues that tests would miss.

Tips:

  • Reproduce the author’s steps. If the description lists a test plan, follow it. Note any missing instructions or surprises.
  • Avoid yak shaving. If setup is taking too long, stop and leave a comment: “I attempted to run the staging environment but hit missing env var XYZ. Can you document setup steps or add a check that fails fast?”
  • Capture empirical feedback. “When clicking Save, I receive a 500; logs show NullPointer on missing profile.id. Likely related to new optional chain paths.”

Your goal is not to be QA. You are adding a small, fast feedback loop that complements unit and integration tests.

Deepening the design and interfaces pass

At the top of your review order, focus on the shape of the change.

Questions to ask:

  • Does the change introduce a new abstraction? Is it the right level? Does it leak details?
  • Are public methods minimal and named for intent rather than mechanism?
  • Are cross-module dependencies moving in the right direction, or is a lower layer calling up into a higher one?
  • Is feature flagging or configuration handled at the edges rather than deep in helpers?
  • Is the change reversible if it needs to be rolled back?

Example interface critique:

  • Before:
    • A function createUser(name, email, sendWelcomeEmail) where the boolean toggles side effects.
  • After suggestion:
    • Separate createUser(userData) from a decoupled sendWelcomeEmail(userId). Or accept an options object with defaults: createUser(userData, { sendWelcomeEmail: false }). This keeps the core action pure and testable.

Public API consideration:

  • If a shared library changes a function signature, look for versioning and deprecation notes. Example: “Renaming fetchUser to getUser breaks downstream. Can we add a deprecated alias and log a warning for one release cycle? Include the migration snippet in CHANGELOG.”

Diagramming helps. A quick sketch in a comment clarifies intent:

  • “Current flow: Controller → Service A → Service B → Repo”
  • “Proposed: Controller → Orchestrator → Service A/B → Repo”
    Encourage the author to add an ASCII diagram or a short doc if the design changed meaningfully.

Checking correctness and tests

Once the design looks sound, verify that the code does what it claims.

  • Trace the main path. Pick a typical input and follow it through. Write down assumptions you see. Example: “Assumes user.locale is set, but we do not default it. What happens for new users?”
  • Look for unchecked return values and swallowed errors. Example: “We catch any exception and return 200. Should failed payment return 4xx with a specific error code?”
  • Verify idempotency where required. Retrying webhooks or payments should not double-charge or duplicate work.
  • Confirm boundary conditions in tests. Tests should cover:
    • Success path
    • Invalid input
    • Error from dependency (timeout, 500)
    • Empty or null data
    • Concurrency if relevant

Concrete test ask with snippet:

Example (JavaScript):

it("retries on 429 with exponential backoff", async () => {
  mockApi.onFirstCall().rejects(rateLimitError());
  mockApi.onSecondCall().resolves({ ok: true });
  const start = Date.now();
  const result = await fetchWithRetry("/data");
  const elapsed = Date.now() - start;
  expect(result.ok).toBe(true);
  expect(elapsed).toBeGreaterThanOrEqual(100); // first backoff
});

If you cannot find such tests, ask for them. If the author argues that existing tests already cover the behavior, ask for pointers to the specific tests and what cases they cover. That teaches both of you where coverage lives and prevents over-testing or duplication.

Call out flaky patterns:

  • Sleeping for fixed time in tests instead of waiting on events or signals
  • Over-mocking internals instead of using public interfaces
  • Tests that assert on logs instead of outcomes

Encourage naming tests by behavior rather than method name. “returns 404 when order not found” is clearer than “testGetOrderNotFound.”

Probing edge cases

Edge cases are often where bugs hide. Think in categories:

  • Input shape and size. Empty arrays, nulls, very long strings, special characters, unexpected enums.
  • Time-related. DST transitions, leap years, time zones, clocks moving backward.
  • Resource limits. Large payloads, many concurrent requests, memory growth on large loops.
  • Permission and roles. Unauthorized access, partial access, feature flags off.
  • Network. Retries, timeouts, idempotency, partial failures.

Targeted questions:

  • “What happens if the webhook fires twice for the same event? Do we dedupe by event id?”
  • “If the batch contains 0 items, do we no-op or write an empty record?”
  • “If a user lacks permission to view subgroup data, do we filter results or reject the request?”

Suggest quick cases to add:

  • “Add test: createUser with email uppercase, ensure stored normalized lowercase.”
  • “Add test: processBatch with an item that fails; verify we skip and continue vs fail all.”

Security and performance quick checks

You may not be the security or performance expert, but a light pass catches common issues.

Security checklist:

  • Input validation and output encoding. Avoid directly inserting user input into SQL, HTML, shell commands.
  • Authentication and authorization checks. Ensure checks align with the feature’s sensitive actions.
  • Secrets handling. No secrets in code, logs, or error messages.
  • Dependency risks. If a new library is added, is it maintained and licensed correctly? Is its use minimal?
  • Data exposure. New fields in API responses or logs, PII masked.

Example comments:

  • “We pass user-provided file names to the OS. Can we sanitize or restrict to a safe whitelist?”
  • “We catch ‘any’ error and return 404. That might hide real server errors; could we map known errors and return 500 otherwise?”

Performance checklist:

  • N+1 queries. Especially in loops or per-item calls.
  • Inefficient data structures. Repeated linear searches when a map would do.
  • Unbounded growth. Caches or lists that never evict, accumulating state.
  • Hot path changes. Code in request handlers, tight loops, or rendering cycles.
  • Logging volume. Debug logs in hot paths can flood storage.

Practical ask with snippet:

Example (Python):

# Current
users = [db.get_user(id) for id in ids]  # N queries

# Suggestion
users = db.get_users_bulk(ids)  # 1 query

If you suspect a hotspot, ask for a quick measurement:

  • “This runs per request. Could you benchmark with 10k items and post rough timings?”
  • “Can we add a histogram metric for handler latency to verify in staging?”

Readability and maintainability

Once you are satisfied with design, correctness, and risks, do a pass for clarity.

Focus points:

  • Naming conveys intent. Prefer “expiresAt” over “t,” “activeDiscounts” over “arr.”
  • Functions are single-purpose and not too long. Extract helpers when a function mixes parsing, validation, and side effects.
  • Comments explain why, not what. Avoid stating the obvious; note non-obvious constraints or tradeoffs.
  • Consistency with project style. If there is a guideline, reference it rather than personal taste.
  • Duplicated logic is consolidated. If two functions differ only slightly, consider unifying.
  • Tests are readable. Arrange tests with clear setup, act, assert sections.

Concrete examples:

  • Instead of:

    // increment
c = c + 1;

    Use:

    // We retry up to 3 times to handle transient 5xx errors
attempts += 1;

  • Instead of a chain of booleans:

    if (isAdmin && !isSuspended && (age > 18 || hasConsent)) { ... }

    Extract:

    const canAccessRestricted = isAllowedRestrictedAccess(user);
if (canAccessRestricted) { ... }

As a reviewer, avoid policing personal style. Tie comments to clarity, maintainability, and team standards.

Writing constructive comments that land

The best comments help the author improve the change now and their future work. Your tone and structure matter as much as your technical content.

Principles:

  • Ask questions before asserting. Start with curiosity to uncover intent: “What tradeoff led you to choose X over Y?” This reduces defensiveness and may reveal constraints you missed.
  • Propose alternatives, not just problems. Offer a concrete path forward when possible.
  • Separate blocking from suggestions. Make it easy for the author to know what must change versus what can be deferred.
  • Include praise. Positive feedback reinforces good patterns and makes tough comments easier to digest.

Comment patterns with examples:

  • Clarifying question:

    • “I see we store retry counts in memory. How do we handle process restarts? Would a persisted counter be safer for jobs that must not exceed N retries?”

  • Suggestion with rationale:

    • “Consider using a map keyed by userId instead of scanning this array each time. This reduces lookup from O(n) to O(1), which matters on the hot path.”

  • Blocking with clear reason:

    • “Blocking: The migration drops a column without a backfill or rollback plan. We need a two-step migration with deploy N adding new column + backfill, deploy N+1 removing the old column after verifying reads are switched.”

  • Non-blocking suggestion:

    • “Nit: We can rename flag to isTruncated for clarity. Non-blocking.”

  • Praise pointing to reuse:

    • “The validatePayload helper is clean and self-contained. Nice job isolating side effects, it made the review straightforward.”

Avoid vague comments:

  • Instead of “This is confusing,” say “I had to jump between three files to follow the order creation flow. Could we centralize orchestration in OrderService so the controller is thin and tests can assert on one place?”

Time-saving templates you can adapt:

  • Intent check: “My understanding is X. I plan to review Y and Z. Please correct me if I missed scope.”
  • Blocking marker: “Blocking due to [reason]. Proposal: [concrete fix].”
  • Alternative: “Another option: [snippet], which [benefit]. Open to other approaches.”
  • Praise: “Strong [specific aspect]. Consider adding this pattern to the guidelines.”

Tone cues:

  • Use “we” for shared goals. “How can we ensure idempotency” reads better than “You did not make it idempotent.”
  • Prefer verbs that invite action. “Could we add a test for negative balance” instead of “No test for negatives.”
  • Keep comments short and focused. If a topic is complex, summarize and suggest a short call or doc rather than packing a thread with back-and-forth.

When to escalate from comments to conversation:

  • If a design disagreement has more than a couple of thread replies.
  • If tone seems to degrade in text.
  • If a change touches cross-team boundaries.

Offer to summarize agreements back into the PR description for future readers.

Examples of constructive comments on small diffs

Small, focused diffs make it easy to suggest concrete changes with snippets.

Example 1: replacing magic numbers

Diff:

- if (attempts > 3) { lockAccount(userId); }
+ if (attempts > 3) { lockAccount(userId); }

Comment:

  • “Consider extracting MAX_LOGIN_ATTEMPTS = 3 to config or a constant. That makes it easier to tune without code changes and clarifies intent.”

Example code offer:

const MAX_LOGIN_ATTEMPTS = 3;
// ...
if (attempts > MAX_LOGIN_ATTEMPTS) lockAccount(userId);

Example 2: pointing to a guideline

Context: Team style guide prefers early returns over deep nesting.

Diff:

if (user) {
  if (user.isActive()) {
    process(user);
  }
}

Comment with reference:

  • “Our style guide recommends early returns to reduce nesting. See ‘Early exits’ section. Suggest:”

    if (!user) return;
if (!user.isActive()) return;
process(user);

Example 3: security precaution

Diff:

logger.info(`User ${userId} failed login: ${reason}`);

Comment:

  • “reason may contain PII or hints useful to attackers. Suggest logging reason code (e.g., INVALID_PASSWORD) instead of raw text. We can keep detailed reason in a secure audit log.”

Example 4: refactor for testability

Diff:

function sendReceipt(orderId) {
  const res = fetch(`.../orders/${orderId}`);
  // ...
}

Comment:

  • “Directly calling fetch in business logic makes testing harder. Consider injecting an httpClient, so tests can pass a mock. For example:”

    function sendReceipt(orderId, { httpClient = fetch } = {}) { /* ... */ }

Using references to anchor feedback

When you cite a rule, link to it. This avoids subjective debates and teaches team norms.

Sources to reference:

  • Team style guide sections (naming, error handling, testing patterns)
  • Architecture decision records (ADRs) for module boundaries or tech choices
  • Security guidelines (secrets handling, auth patterns)
  • Performance playbooks (caching, batching, logging)
  • Language or framework docs (official recommendations)

Concrete pattern:

  • “Per ADR-017, repositories should not depend on services. The new Repo calls AlertService. Could we invert this dependency by moving alerting to the service layer?”

If a reference does not exist, propose creating one. “We have discussed early returns in multiple PRs. Shall we add a short section to the style guide? I can draft it based on this example.”

Managing time and energy in reviews

Reviews are thinking work. Attention fades. Manage it like any other limited resource.

  • Timebox sessions. Set 30–45 minute blocks. When the timer ends, either wrap or take a short break. Long stretches lead to sloppy comments and missed risks.
  • Start with high-impact parts. Use the diff map to pick core files or public interfaces first. Leave cosmetics for the end, or not at all.
  • Avoid fatigue-driven nitpicks. If you notice yourself commenting on commas or small spacing, you might be tired. Stop, or leave one summary note: “Let’s run the formatter and address remaining nits after main feedback.”
  • Batch similar comments. Instead of repeating “rename idx to index” 12 times, write one general comment and ask the author to apply it across the PR.
  • Ask to split large changes. If your first pass suggests the PR is too large to review thoroughly, say so early. “This is 1,800 lines across 25 files. To give it the attention it needs, could we split migrations and feature logic into separate PRs?”
  • Use checklists. A small, personal checklist reduces cognitive load. Example:
    • Read description
    • Skim file list
    • Design and interfaces
    • Correctness and tests
    • Edge cases
    • Security and performance
    • Readability
    • Summary comment and labels

Balance speed and quality:

  • If the change is low risk (docs, comments, small refactor), skim and approve quickly.
  • If it is medium risk, do the full process but keep comments concise.
  • If it is high risk, go deeper, ask for tests upfront, and get a second reviewer.

Dealing with large or messy changes

Sometimes you face a PR that is too big or mixes many concerns. Here is a constructive path forward:

  • Name the problem neutrally. “This PR tackles feature A, refactors module B, and updates CI. Reviewing all at once makes it hard to verify each piece safely.”

  • Propose a split plan:

    • PR 1: Test-only safety net for existing behavior (if feasible)
    • PR 2: Infrastructure changes (CI, config)
    • PR 3: Pure refactor with no behavior change
    • PR 4: Feature implementation behind a flag
    • PR 5: Flag flip and cleanup

  • Suggest mechanical separation. “Can you send a formatter-only commit separately? That will make the diff signal clearer.”

  • Use labels or draft state to stage review. “Mark the PR as draft until tests land. I will review design now and return for correctness once tests are added.”

  • Encourage incremental merges. “Merge the non-risky refactors first to reduce the diff. Then the feature PR will be smaller, and test diffs clearer.”

  • Offer to pair. For complex rewrites, a short call to align on structure can save days of back-and-forth.

Calibrating severity: blocking vs suggestions

Be explicit about what gates approval. Ambiguity causes churn.

  • Mark blocking items at the top in a summary comment. “Blocking: 1) missing test for retry backoff, 2) unsafe logging of Authorization header.”
  • Use a consistent tag in line comments. Prefix with “Blocking:” or “Non-blocking:” so authors can filter and address accordingly.
  • Limit blocking to issues that risk correctness, safety, or maintainability at scale.
  • Move stylistic or minor improvements to suggestions. If they are important, link to a guideline and ask to follow it globally, not just in the PR.

Example summary:

  • “Summary:
    • Blocking:
      • Add test for DST boundary in parseSchedule
      • Sanitize headers before logging
    • Suggestions:
      • Extract calculateDiscount into its own module
      • Rename fn run to process
  • “Once the two blocking items are addressed, I am comfortable approving.”

Examples of asking questions, proposing alternatives

Sometimes the best review nudge is a question that points to a better path.

  • Question opening a design door:

    • “What led you to choose inheritance here instead of composition? Could composition reduce the coupling to BaseProcessor?”

  • Short alternative with benefit:

    • “Alternative: use a strategy map:”

      const handlers = { flat: flatDiscount, tiered: tieredDiscount };
handlers[type](...args);

      This removes the switch and makes adding a new type simpler.

  • Question surfacing a rollout concern:

    • “If we roll this out gradually, how do we handle data created before the migration? Do we need a read path that supports both schemas during rollout?”

  • Alternative for safer migration:

    • “Two-step migration: add new column, dual-write, backfill in background, switch reads, then drop old column in a later PR. This reduces downtime and rollback complexity.”

Using examples and concrete snippets in comments

Concrete changes carry more weight than abstract advice. When suggesting improvements:

  • Show a minimal diff. Keep it small and focused on the idea.
  • Respect the author’s style and context. Keep changes close to what is already there.
  • Clarify what problem the snippet solves.

Example: replacing try/catch sprawl

Before:

try {
  const user = await getUser(id);
  if (!user) throw new Error("no user");
  const profile = await getProfile(user.profileId);
  if (!profile) throw new Error("no profile");
  // ...
} catch (e) {
  logger.error(e);
  return res.status(500).send("error");
}

Suggestion:

const user = await getUserOr404(id);
const profile = await getProfileOr404(user.profileId);
// ...

Comment:

  • “By moving not-found handling into helpers that return 404, we keep the main flow straight and avoid masking 404s as 500s.”

Point to a pattern:

  • “We have getOrCreate in UserRepo that follows this pattern. Using it here would reduce duplicate logic.”

Bringing in external references judiciously

Citing external sources can help, but avoid overloading the author.

  • Link to a specific section, not a long document. Quote the relevant lines.
  • Align with team norms. If an external best practice conflicts with local constraints, acknowledge that.
  • Offer to adapt the reference into a local guide. “This testing table pattern maps well to our case. If we like it, I can add a short example to our docs.”

Example reference comment:

  • “OWASP recommends output encoding for any user-provided data inserted into HTML. See ‘XSS Prevention Cheat Sheet Step 5’. In our template, we are interpolating raw user.bio. Suggest using the escape helper: {{ escape(user.bio) }}.”

Closing the loop: summarizing and approving

After leaving line comments, add a short summary that:

  • Restates your understanding of intent
  • Highlights any blocking items
  • Commends what is strong about the change
  • Notes next steps or who else should review if needed

Example:

  • “Intent: add retry with exponential backoff for 5xx responses on fetchData. Design looks good; retry logic isolated in RetryPolicy. Blocking: add a test for jitter and ensure we cap attempts at 5. Non-blocking: rename delayMs to backoffMs and add docstring with formula. Strong work on isolating policy from transport.”

When the author addresses feedback:

  • Re-review only the updated areas if possible.
  • Acknowledge fixes explicitly. “Thanks for adding the DST tests and masking headers. Approving.”
  • If a suggestion is deferred, agree on a ticket or follow-up PR. “Created ticket DEV-123 to track refactor to strategy map; let us not block this feature on it.”

Handling disagreements constructively

You will disagree. Keep it professional and focused on outcomes.

  • Seek shared goals. “We both want a safe rollout and maintainable code. Given that, which option reduces risk the most in the next two weeks?”
  • Offer tradeoffs, not ultimatums. “Option A is simpler now but risks a data migration later; Option B is more work now but keeps the repo pure. My lean: B, because this repo is used by three services.”
  • Bring data. “We saw incidents last quarter from tight coupling in repos. Keeping repos pure reduced review time in the payments team.”
  • Defer to guidelines when they exist. If none exist, propose writing one after the immediate decision.
  • Know when to accept. If both options are valid and time is tight, prefer unblocking with a follow-up task for improvements.

If the thread grows long, suggest a short call. Afterward, summarize decisions and rationale in the PR for posterity.

Reviewing non-code artifacts: configs, migrations, infra

Many risky changes live outside pure code.

  • Config changes. Check defaults, validation, and rollout plan. “New feature flag default is true in production. Is that intended? Should we default to false and flip per environment?”
  • Database migrations. Look for backward compatibility, locking risks, and rollback strategy. “Adding a NOT NULL column without default will lock writes. Suggest adding with default, backfilling in batches.”
  • Infrastructure as code. Confirm drift, permissions, and blast radius. “This policy grants s3:ListAllMyBuckets. Is that broader than required? Could we scope to the specific bucket ARN?”
  • CI/CD scripts. Verify secrets, caching, and failure modes. “We added npm ci without cache restore; build times will spike. Can we restore cache keyed by lockfile hash?”
  • Documentation. Ensure user-facing changes have docs updates. “We added a new CLI flag; can we update the README and example commands?”

Ask authors to include a test plan or verification steps for these changes. For migrations, request a dry run on staging and metrics to watch during rollout.

Practical security and performance questions to copy-paste

When in doubt, these quick prompts surface issues fast:

Security:

  • “Are all inputs validated at the boundary?”
  • “Could any logged data include PII or secrets?”
  • “Are authz checks at the right layer and consistent with policy?”
  • “Does this introduce a new dependency? Is it vetted?”
  • “Could error messages leak sensitive details?”

Performance:

  • “Any N+1 patterns or per-item remote calls?”
  • “Is this on a hot path? If so, any quick benchmark?”
  • “Do we allocate unnecessary objects in loops?”
  • “Is there unbounded growth in in-memory state?”
  • “Any expensive logs or debug statements in production paths?”

Leveling up readability: what to look for

Beyond style, readability is about telling a clear story.

  • Structure code so the main path is first, details later. Early returns and helpers make this possible.
  • Prefer data structures that match the domain. Use named records or small value objects instead of raw tuples or arrays.
  • Keep tests narrative. Use descriptive test names, clear setup, and helper factories to reduce noise.

Before:

test("1", async () => {
  const a = await u();
  const b = await v(a);
  expect(b[0]).toBe(3);
});

After:

test("returns 3 retries on first 429", async () => {
  const user = await createUser();
  const result = await fetchWithRetry(user.request);
  expect(result.retries).toBe(3);
});

Offer to extract repetitive test setup into builders or fixtures when it obscures assertions.

Encouraging positive patterns with praise

Do not only point out problems. Noticing good work shapes habits.

  • Be specific. “Great separation between parsing and validation in ConfigLoader. It made edge case reasoning simple.”
  • Connect to impact. “The choice to make RetryPolicy pure allows us to reuse it in the scheduler. That will save time on the next feature.”
  • Suggest documenting good patterns. “This structured log field naming is clean. Let us add this example to our logging guide.”

Praise is not fluff; it is guidance. Authors repeat what gets recognized.

Bringing it all together on a real change

Imagine a PR that:

  • Adds a RetryPolicy abstraction
  • Updates fetchData to use retry on 5xx
  • Adds a config flag for max attempts
  • Touches logging to add context
  • Includes two new unit tests

Your review flow:

  • Context. Read description. It says: “Handle transient 5xx with retries, default maxAttempts=3, jitter backoff.” Tests listed. Good.
  • Skim diff. Files: retry.js, fetchData.js, config.ts, logger.ts, tests. Manageable size.
  • Design. RetryPolicy as a pure function returning nextDelay and whether to continue. Good separation. fetchData depends on it via an injected dependency. Good.
  • Correctness and tests. Tests cover retry after first 5xx and stop on 4xx. Ask for test on jitter and cap at max attempts. Suggest test for non-idempotent methods skipping retry.
  • Edge cases. Ask about 429 (rate limit) handling and idempotency for POSTs with side effects.
  • Security/performance. Logging includes status and attempt count; ensure no token in logs. Performance: no heavy operations introduced; backoff delays are correct.
  • Readability. Rename delayMs to backoffMs, add docstring to RetryPolicy on formula, move constants to config with sane defaults.

Your summary:

  • “Intent: add retry with jitter for 5xx in fetchData. Design: clean isolation via RetryPolicy. Blocking: add test for jitter distribution and enforce no retry on non-idempotent methods unless explicitly allowed. Suggestions: rename delayMs, doc comment on formula, mask any auth headers in logs. Nice work introducing a pure policy with injected dependency.”

The author adds tests and a small guard for idempotency. You approve and leave a final note praising the isolation and proposing adding RetryPolicy to shared utilities later.

Growing your personal review practice

Make your process explicit and iterate.

  • Keep a short personal checklist handy. Paste it atop your first comment until it becomes muscle memory.
  • Track your top three recurring comments. If you repeat them across PRs, propose a guideline or a linter rule.
  • Reflect weekly on a review you are proud of and one you are not. What made the difference? Timeboxing, clarity, or tone?
  • Ask for feedback on your reviews. “Was my summary helpful? What could I do to make my comments more actionable?”
  • Pair-review occasionally. Watching a teammate’s process can reveal blind spots and new heuristics.

Consistency does not mean rigidity. Adapt depth and focus to the change’s risk. But keep the order: start with context, review design first, verify correctness with tests, probe edges and risks, then polish readability. Use clear, kind, and actionable comments. Offer examples. Manage your time. And remember, your goal is not to win arguments, but to help the team ship safer, simpler code.
